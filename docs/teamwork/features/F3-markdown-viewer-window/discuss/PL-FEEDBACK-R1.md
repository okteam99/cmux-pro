# F3 · PL Feedback · R1

## 总体评价

条件通过，但**不建议作为当前迭代的高优先级**交付。PRD 本身结构完整、AC 较细，属于合格草稿；但从产品定位看，这是一个「给终端工作台加一个体积颇大的文档阅读器」的功能，边际价值不高、体积代价较大（~4MB bundle），且与系统自带/第三方 md 阅读器存在明显的竞争重叠。主要顾虑三点：（1）新窗口版与 tab 版并存缺少收敛路线，会让同一个"打开 md"的交互产生两种心智模型；（2）Mermaid 是真实刚需但用户规模窄，是否必须本地打包完整 mermaid.js 值得再评估；（3）PRD 没有清晰回答"为什么用户要在 cmux 里读 md，而不是用 Typora / 系统 Quick Look / VS Code"，缺失竞争定位与目标用户画像。

## 逐项反馈

### 1. 产品定位 & 战略

- cmux 的核心定位是**终端工作台**（终端 + 分屏 + 工作区），文档阅读是辅助能力。本 PRD 把 md 阅读器做成"独立 NSWindow"后，它与 cmux 主窗口在心智上已经解耦——那用户为什么不直接用更专业的 md 阅读器？这会弱化 cmux 的"一站式"卖点（一站式价值来自"文档和终端在同一工作区视野内"，而独立窗口恰好削弱了这一点）。
- 建议 PRD 补充一段**竞争定位**：本 viewer 相对于 Quick Look（空格键预览 md）、Typora、Obsidian、VS Code 预览有什么差异化？目前读下来的"差异"只是 Mermaid 支持，这个差异点很薄，且其他工具都能做到。
- 战略上更自然的路径其实是**加强 tab 版** MarkdownPanel 使其支持 Mermaid，让 md 阅读成为工作区内的一等公民；而不是新开窗口变成脱离工作区的弱独立产品。PRD 选择了相反的方向，需要论证。

### 2. 用户价值 & 优先级

- **目标用户缺失**：PRD 第 2 节用户场景列了 5 个场景但没指出 TA。是"阅读开源项目 README 的用户"？"写架构文档的 tech lead"？前者 Quick Look 就够，后者更可能用 Typora/Obsidian 深度编辑。
- **ROI 偏低**：对比同期候选（终端性能、稳定性 bug、协作/分享），本功能属于"锦上添花"。而 PRD 引入的成本是：4MB bundle、新窗口生命周期管理代码、第二条 md 渲染链路的长期维护成本、JS 依赖的 CVE 跟进负担（mermaid/marked/highlight 都会有安全更新）。
- **Mermaid 需求强度未量化**：PRD 说"cmux 项目内外 md 广泛使用 Mermaid"，但没有用户访谈/埋点数据支持。建议先做轻量调研（issue 搜索、用户反馈条数）再定优先级。

### 3. 范围边界 & 风险

**3.1 新窗口 vs tab 版并存的收敛问题**
- PRD 1.2 说"原有 MarkdownPanel tab 版保留不动…后续可收敛，本次不涉及"——这是**把技术债明确写进 PRD**。必须在本轮就回答：默认入口走哪个？是否保留双入口？长期谁是主战场？否则 File Explorer 的 md 双击行为一旦改为新窗口，老用户的 tab 习惯会被破坏；而如果保留两种入口，UI 又变复杂。
- 建议 PRD 明确"收敛时间表"：要么本期就下掉 tab 版（推荐），要么本期新窗口只作为"另存入口"（例如 "Open in Viewer Window" 菜单项），File Explorer 双击仍保持 tab 行为不变。

**3.2 4MB bundle 增量**
- 对一个 macOS 原生终端应用，为了"偶尔看带 Mermaid 的 md"多装 4MB 基本可接受，但并非显然划算。可考虑：
  - 按需下载 mermaid.js（首次打开含 mermaid 的 md 时提示下载）——体积 0，但有离线/启动复杂度代价。
  - 仅打包 marked + highlight（~750KB），Mermaid 通过渲染 ASCII 占位 + 按钮"点击在浏览器渲染"——体积显著下降。
  - 直接不做 Mermaid，改为调用系统默认应用打开——体积 0。
- PRD 给出的是"最贵的方案"但没有论证为什么必须是最贵的。

**3.3 与系统默认的竞争定位**
- AC-18「相对路径链接 `./foo.md` 用 NSWorkspace 打开系统默认」这一行事实上承认了"系统默认也能看 md"。那么反过来问：既然系统默认够用，为什么 cmux 点击 md 不直接走系统默认？PRD 需要回答这个问题，否则 AC-18 的设计本身就在暗示本功能是冗余的。

**3.4 生态 / 协作缺口**
- 开发者读文档的高频动作是"把这段 md/图分享给同事"。PRD 完全没有涉及：导出为 PNG/SVG、复制 Mermaid 图、"复制渲染后文本"等。如果本功能希望长期存在，至少应为"导出 Mermaid SVG"留接口（非目标里明确排除导出 PDF，但没排除导出 Mermaid 图——这是一个差异化机会点）。
- 无协作/同步阅读相关讨论。短期不需要，但 PRD 的"非目标"里应明确写出"不支持共享 / 协同阅读"，避免后续边界漂移。

**3.5 Mermaid 版本锁死风险**
- 锁 Mermaid 10.x LTS 是对的，但 CVE 跟进责任应落到某个角色（运维 / RD 谁负责），PRD 未写。

### 4. PRD 完备性

- **缺失性能 AC**：没有定义"窗口从点击到首屏渲染完成"的时间预算（首次加载 WKWebView + mermaid.js 可能 500ms+，需要有上限）。
- **AC-21"文件过大 > 1MB 正常加载不做截断"风险较大**：10MB 的 md + 多个 Mermaid，WKWebView 可能卡 UI。建议改为"定义阈值 + 超阈值时显示 loading 或降级"。
- **AC-4 窗口记忆策略歧义**："关闭后记忆；下次打开（不同文件亦可）尊重该记忆"——是每个文件单独记忆，还是全局共用一份？PRD 自己在第 9 节也标注这点待用户确认。
- **AC-19 文件监听刷新 1-2s**：如果用户正在滚动到中间位置，刷新后是否保持滚动位置？没有定义。这是很常见的体验陷阱。
- **AC-12 Mermaid 错误提示**："代码块处显示错误提示"需要指明错误文案本地化（当前 PRD 只在第 4 节模糊提到新增一个本地化 key）。
- **埋点 DEBUG only**：`markdownViewer.open` 等事件只在 DEBUG 记录，则产品侧永远拿不到生产数据来验证 ROI。建议至少核心事件（open 计数、mermaid 是否被用到）接入生产埋点（若 cmux 有匿名遥测管道）。
- **非目标"不支持 md 编辑"应明确原因**：避免后续被要求"加一个编辑模式"时边界模糊。
- **风险第 8 节**没覆盖：WKWebView 内存占用（每个窗口一个 web view，多文档并读时内存）、Mermaid 大图导致的 SVG 过大导致滚动卡顿、JS 第三方依赖的供应链安全。

### 5. 建议分歧项（需用户裁定）

1. **战略方向裁定**：是做"独立窗口 md viewer"（当前 PRD 方向），还是"强化 tab 版 MarkdownPanel 支持 Mermaid"（替代方案）？这是 PRD 根基，需用户拍板。
2. **bundle 体积方案裁定**：全量本地打包 mermaid（4MB）/ 按需下载 / 放弃 Mermaid 走系统默认——三选一。
3. **tab 版收敛时间表**：本期下掉 tab 版？并存？还是相反——File Explorer 默认仍走 tab 版，新窗口作为可选入口？
4. **优先级裁定**：是否把本功能放到当前迭代？若排入，需明确牺牲哪个其他候选功能。

## 收敛建议

- **必改项**：
  - PRD 补充"目标用户画像 + 竞争定位"（vs Quick Look / Typora / VS Code 预览）章节，回答"为什么在 cmux 里做 md 阅读器"。
  - PRD 明确新窗口 vs tab 版的**收敛路线和默认入口**，消除"双栈并存"技术债。
  - PRD 补充**性能 AC**：首屏渲染时间、文件大小阈值、滚动位置保留（AC-19 衍生）、内存上限（多窗口）。
  - PRD 修正 AC-4 窗口记忆策略（每文件 vs 全局），本轮就定，不留 "待用户确认"。
  - PRD 在"非目标"补写：不支持共享/协同阅读、不支持编辑模式（并写明原因）。
  - Mermaid 依赖打包方案需要在"风险"节给出量化权衡，而非直接接受 4MB。

- **可讨论项**：
  - 导出 Mermaid SVG/PNG 是否作为 P1（差异化机会）。
  - 核心事件是否接入生产埋点（非 DEBUG）以验证 ROI。
  - Mermaid CVE / 依赖更新的 owner 归属。
  - AC-18 相对路径链接的行为是否应改为"在 viewer 内打开"以形成闭环（反向意见：会增加状态机复杂度）。

- **无异议项**：
  - 基础 md 渲染 AC（AC-7、AC-13、AC-15、AC-16、AC-17）合理。
  - JS 本地化引入、不走 CDN（离线保证）合理。
  - 外部链接走 `NSWorkspace.shared.open` 合理。
  - 文件监听沿用 MarkdownPanel watcher 的语义合理。
  - 锁定 Mermaid LTS 版本合理。
