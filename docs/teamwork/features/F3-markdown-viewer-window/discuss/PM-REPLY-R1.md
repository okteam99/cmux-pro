## 对 PL 反馈的逐条回应

### PL-1 产品定位与竞争差异化缺失
- **态度**：CLARIFY（部分）+ REBUT（部分）
- **回应**：
  - REBUT 「独立窗口削弱一站式卖点」：一站式 ≠ 所有东西塞进一个窗口。cmux 的一站式价值是**环境（CWD / Git ctx / 工作区语义）一致性**，不是**物理上同屏**。独立窗口仍由 cmux 唤起、保持文件监听、跟随 cmux appearance，不妨碍"工作台"定位，反而解决了「文档内容与终端挤 tab」这一真实痛点（PRD 1.1 已列）。
  - CLARIFY 「为什么不用 Typora / Quick Look / VS Code」：用户明确要求在 cmux 内处理 .md 打开流（File Explorer 是 cmux 一等入口），Quick Look 不支持 Mermaid、Typora/VS Code 是外部应用需上下文切换。但 PL 的建议合理——PRD 应明确写出竞争定位。
- **PRD 变更**：在第 1 节后新增 **1.4 竞争定位与目标用户**，列出 vs Quick Look / Typora / VS Code 的差异，并标注目标用户=cmux 重度使用者中阅读项目内 .md 的场景。

### PL-1（续）战略方向应反过来：强化 tab 版支持 Mermaid
- **态度**：DIVERGE
- **回应**：PL 建议「强化 tab 版支 Mermaid」是合理的技术路径，但用户的明确需求是**独立窗口**（"文档与终端挤 tab 区"是痛点本身）。把 Mermaid 塞进 tab 版不能满足用户诉求。两种选择代价：
  - 按 PRD（独立窗口 + WebView）：满足用户指令；引入 ~4MB bundle、两套渲染栈短期并存。
  - 按 PL 建议（tab 版加 Mermaid）：省 bundle / 无双栈；但不满足用户原始需求，需要用户重新确认是否放弃"独立窗口"诉求。
- **裁定需求**：已由用户裁定为独立窗口方向，但 PL 意见会记录进 DIVERGE 清单供用户二次确认。

### PL-2-1 目标用户缺失
- **态度**：ACCEPT
- **回应**：PRD 第 2 节确实只列场景未定 TA。
- **PRD 变更**：新增 1.4 节中明确 TA = "cmux 重度使用者，需要在项目仓库内阅读含架构图 / 流程图的 .md 文档（README、ARCHITECTURE、RFC、设计文档）"。

### PL-2-2 ROI 偏低
- **态度**：REBUT
- **回应**：本功能为用户直接指令，ROI 判断由用户自己做出。PL 提到的"4MB bundle + 新窗口生命周期 + 第二条渲染链路 + CVE 跟进"都是真实成本，但用户已评估后下达指令。PM 不在此重新打开立项讨论。

### PL-2-3 Mermaid 需求强度未量化
- **态度**：REBUT
- **回应**：用户是 cmux 最重度的使用者，其自用频率 + 反馈即是最强信号（cmux 当前是小团队产品，没有生产遥测管道做大规模量化）。此外，本仓库 `docs/teamwork/` 下已大量使用 Mermaid（符合 PRD 1.1 陈述），项目内自证。

### PL-3-1 新窗口 vs tab 版收敛路线 / 默认入口
- **态度**：ACCEPT（部分）+ DIVERGE（部分）
- **回应**：PL 说得对，PRD 把"后续可收敛"写进去是明显的技术债。本期我主张：**File Explorer 默认入口改为新窗口**（匹配用户需求），**tab 版保留但不再主动创建入口**（逐步自然废弃）。这样避免本期就删老代码引入风险，但默认入口唯一不造成心智分裂。
  - 另一选项（PL 推荐）：本期就下掉 tab 版。代价：删除既有 MarkdownPanel 代码 + 回归测试；好处：零双栈。
- **PRD 变更**：1.2 节改写"tab 版保留"条款为「tab 版本期冻结（不再暴露新入口），下一迭代视新窗口稳定度决定下线」；在 DIVERGE 中保留"本期立刻下线 tab 版"作为用户备选。

### PL-3-2 4MB bundle 增量未做方案权衡
- **态度**：ACCEPT
- **回应**：PRD 第 8 节确实直接写"接受"没论证。PL 给的三选项有价值，但"按需下载"会破坏离线保证（违背 PRD 8 节离线原则，用户是离线/安全敏感用户）；"不做 Mermaid"违背用户指令。本 PRD 仍选"全量本地打包"，但应把权衡矩阵写进 PRD 让决策可审计。另外可做**可选优化**：mermaid.js 使用精简构建（仅 flowchart/sequence/class/gantt，去 ER/C4/Mindmap 等），可压到 ~1.5-2MB。
- **PRD 变更**：第 8 节「bundle 体积」扩写为方案对比表（全量/按需/放弃 Mermaid/精简打包），并说明选择"精简打包"作为 P0，"全量 4MB"作为 fallback。

### PL-3-3 AC-18 相对路径走系统默认 = 承认系统默认够用
- **态度**：REBUT
- **回应**：不同意这个推理。AC-18 走系统默认是为了**避免跳转 viewer 的状态机复杂度**（多文件堆栈、返回按钮等），而不是"系统默认够用就该放弃本功能"。单文件场景（打开 README.md 一次读完）系统默认不够用的痛点恰恰是 Mermaid 不支持，这正是本功能的价值点。
- **PRD 变更**：AC-18 增加一条注释，说明选择"系统默认"是**避免 viewer 变成浏览器**的刻意设计，不是能力缺陷。

### PL-3-4 生态/协作缺口（导出 Mermaid SVG/PNG、共享、协同阅读）
- **态度**：ACCEPT（非目标补全）+ CLARIFY（导出为 P1 留接口）
- **回应**：PL 观察到位。"导出 Mermaid SVG"作为差异化机会点值得留接口，但本期不做；协同/共享本期明确列入非目标。
- **PRD 变更**：
  - 1.3 非目标新增：「不支持共享 / 协同阅读」「不支持 md 编辑（原因：viewer 定位只读、编辑器生态已充分，cmux 不进入该赛道）」「不支持导出 PDF；导出 Mermaid SVG 留 P1 接口，本期不实现」。
  - 5 节接口变更新增：WebView 侧 `cmux.exportMermaidSVG(nodeId)` 预留（本期不实装，仅文档占位）。

### PL-4-1 缺失性能 AC
- **态度**：ACCEPT
- **回应**：首屏时间必须量化。
- **PRD 变更**：3.x 新增 AC-23「首次打开 md 窗口 → 基础 md 首屏渲染 ≤ 500ms（冷启动 WKWebView + 加载本地 JS 不计入首次 warm-up）；再次打开（WebView 进程已 warm）≤ 200ms」；AC-24「Mermaid 图单图渲染 ≤ 300ms（中等复杂度 < 30 节点）」。

### PL-4-2 AC-21 大文件无上限
- **态度**：ACCEPT
- **回应**：无上限确实风险大。
- **PRD 变更**：AC-21 改写为「文件 ≤ 2MB 直接渲染；2-10MB 显示加载指示器并渲染；> 10MB 仅渲染前 10MB 并顶部提示『文件过大已截断』」。

### PL-4-3 AC-4 窗口记忆粒度歧义
- **态度**：ACCEPT
- **回应**：本轮就定：**全局共用一份**（初始 900×700 居中；关闭后记忆最后一个 viewer 窗口的大小/位置；新窗口沿用）。每文件独立记忆对 viewer 来说过度设计，用户多数场景是「依次读几个文档」。
- **PRD 变更**：AC-4 明确写"全局共用"；删除第 9 节"AC-4 待用户确认"。

### PL-4-4 AC-19 刷新后滚动位置保留
- **态度**：ACCEPT

### PL-4-5 AC-12 Mermaid 错误文案本地化
